** DONE 001 — =001-greet.py= — plain function with args and return

File: [[file:001-greet.py][001-greet.py]]

*** Presented (v1)

#+begin_src python
"""Small greeting helper — nitpick fodder for snippet 001."""
from typing import Optional


def greet(name: str, greeting: Optional[str] = None) -> str:
    """Return a greeting for the given name.

    Args:
        name: Person to greet.
        greeting: Optional custom salutation. Defaults to "Hello".

    Returns:
        The formatted greeting.
    """
    if greeting is None:
        greeting = "Hello"
    return f"{greeting}, {name}!"


if __name__ == "__main__":
    print(greet("Nathan"))
    print(greet("Nathan", greeting="Howdy"))
#+end_src

*** Feedback

Nathan: "So this is all very verbose. Especially the docstring, it's
very googley. I really like the way clojure has done things:
[[https://github.com/clojure/clojure/blob/clojure-1.12.4/src/clj/clojure/core_print.clj#L16][clojure/core_print.clj]]"

Reading the referenced file: public vars get short *prose* docstrings
that describe behavior, defaults, and edge cases — no =Args:= /
=Returns:= / =Raises:= templates. Internal helpers
(=print-sequential=, =print-meta=, =print-object=, etc.) carry *no*
docstring at all; name + inline code is the documentation. Names of
well-understood parameters are single-letter (=w=, =o=, =xs=).
Structure is dense; no ceremony.

*** Revised (v2)

#+begin_src python
def greet(name: str, greeting: str = "Hello") -> str:
    return f"{greeting}, {name}!"
#+end_src

*** Rules canonized from this snippet

1. *Docstrings are prose, not templates.* No =Args:= / =Returns:= /
   =Raises:= section headers (Google / NumPy / Sphinx styles are out).
   When a docstring exists, it's one or more complete sentences
   describing what the thing does and any non-obvious behavior.

2. *No docstring on small, self-explanatory functions.* Rely on name +
   type hints. Docstrings are for public API surface and for any
   behavior that can't be read off the signature.

3. *No module docstrings on snippet files.* The filename and context
   already say what the file is.

4. *Literal default values in the signature* when the default is
   immutable and cheap (=greeting: str = "Hello"=). Reserve the
   sentinel-and-reassign pattern (=x=None= then
   =if x is None: x = ...=) for cases where the default is mutable or
   has to be computed at call time.

5. *No =if __name__ == "__main__":= demo block on snippets.* A snippet
   is a thing to read, not a thing to run. Demos belong in tests or a
   REPL, not wedged into the file.

*** Still open

- *Union-with-=None= syntax* — the PEP 604 =str | None= vs
  =Optional[str]= question never got to exercise its muscle because
  the revised snippet has no nullable arg. Deferred to a later snippet.
- *Type-hint scope* — the revised version keeps full annotations on
  both args and the return. Clojure's precedent would argue for
  annotating only where it clarifies (analogous to =^Writer= interop
  hints on specific args). Still a real question: do we annotate all
  public function signatures by default, or only when the types aren't
  obvious from the name? Deferred to a snippet where the choice is
  more consequential.

** DONE 002 — =002-find-first.py= — iteration, sum-type result

File: [[file:002-find-first.py][002-find-first.py]]

*** Presented (v1)

#+begin_src python
def find_first(items: list[str], query: str) -> str | None:
    for item in items:
        if item == query:
            return item
    return None
#+end_src

*** Feedback on v1

Nathan:

- =None | str= ordering inverts the meaning to "usually nothing but
  sometimes str" — misrepresents the function.
- "Architecting around =Optional= like this is already a code smell
  because it's utilizing =None= as an overload of an end flag rather
  than using a proper type that defines that something has ended."
- "If I am going to bother with type hints in a dynamic language I
  want EXPLICIT types that tell the whole story and don't leave
  things up to guesswork like pythonistas (read: googlers) are so
  willing to do."
- Prefers =next()= over a generator for this snippet specifically.

Underlying argument: =None= is already saturated with language-level
meanings (default arg, no-return, uninitialized, cache miss, etc.);
reusing it as a domain "not found" sentinel layers yet another
meaning on an already-overloaded symbol — C's =NULL= doing every
absence-meaning at once.

*** Revised (v2) — wrong turn

#+begin_src python
class NotFound(Exception):
    pass


def find_first(items: list[str], query: str) -> str:
    try:
        return next(x for x in items if x == query)
    except StopIteration:
        raise NotFound(query) from None
#+end_src

*** Feedback on v2

Nathan: "This specific example find-first.py is WRONG. Because it is
overloading python's exception system with an improper
responsibility: signal flagging. It is not an EXCEPTION to not find
something you are searching for, that is a legitimate output and
should be reflected in the type itself, not thrown as an error.
This sort of mistake of overloading an exception system is exactly
what leads people to golang's foolishness of removing exception
systems completely, which is also wrong."

The sharpened principle: exceptions are for *actually exceptional*
conditions (I/O failure, contract violation, programmer error,
unrecoverable state). "The thing I searched for wasn't there" is a
legitimate, design-time-expected outcome of a search. It belongs in
the *type*, not in a thrown exception. Go's answer (delete exceptions
entirely because people abuse them) is also wrong; exceptions are
the right tool for truly exceptional conditions and should be kept
pure for that purpose.

*** Revised (v3) — also wrong turn

#+begin_src python
from dataclasses import dataclass
from typing import Generic, TypeVar

T = TypeVar("T")


@dataclass(frozen=True)
class Found(Generic[T]):
    value: T


@dataclass(frozen=True)
class NotFound:
    query: str


def find_first(items: list[str], query: str) -> Found[str] | NotFound:
    return next((Found(x) for x in items if x == query), NotFound(query))
#+end_src

*** Feedback on v3

Nathan: "This syntax APPEARS elegant at first glance but upon deeper
inspection reveals that we have complicated the CALLER location for
the sake of FAKE TYPES. Which boils down to the fact that we have
complicated code path for the sake of documentation. Which is
extremely incorrect."

Then: "Holy crap dude it's just =str | NotFound= that preserves the
happy path while also structurally informing the caller for the not
found path instead of an opaque None type."

The problem with v3 was wrapping the /happy/ path in =Found=. At
runtime =Found= does nothing — it's a dataclass the interpreter
ignores. But it forces every caller to unwrap on every use, even
callers who just want to pass the value through. That's caller
ceremony bought purely for a type-checker label. This is the
Python-specific critique: in Rust where sum types compile to
efficient tagged unions and the compiler enforces exhaustive matching,
wrapping the happy path pays off. In Python, it doesn't.

*** Revised (v4)

#+begin_src python
from dataclasses import dataclass


@dataclass(frozen=True)
class NotFound:
    query: str


def find_first(items: list[str], query: str) -> str | NotFound:
    return next((x for x in items if x == query), NotFound(query))
#+end_src

- Happy path is plain =str=. Caller who just wants the value uses it
  directly — no =.value= unwrap, no =match= statement, no
  =isinstance= branch.
- Failure path is a named domain type =NotFound=, not =None=. Carries
  the =query= it couldn't find — debuggable, not an opaque flag.
- Caller who /does/ need to branch checks =isinstance(result,
  NotFound)= (or the inverse on =str=). Only the branching caller
  pays; the pass-through caller doesn't.
- Body is one line: =next(gen, default)= with the domain type as the
  default. Zero exception handling.

*** Rules canonized from this snippet

1. *Don't use =None= as a domain absence sentinel.* =None= is already
   saturated with the language's absence meanings; layering a domain
   "not found" meaning on top repeats C's =NULL= mistake.

2. *Don't use exceptions for expected-absence outcomes.* Exceptions
   are for actually-exceptional conditions (I/O failure, contract
   violation, programmer error). "Not found when searching" is a
   routine outcome, not an exception. Go's answer (delete exceptions
   entirely) is also wrong — keep the exception channel pure so it
   retains meaning.

3. *Union with a named absence type keeps the happy path unwrapped.*
   Return =T | NamedFailure= — not =Found[T] | NotFound=. The
   happy-path type is used directly by callers who don't need to
   branch; only branching callers pay the =isinstance= cost.
   =NamedFailure= should carry context (what was searched for, what
   condition failed).

4. *In other languages where sum types are compile-time-enforced and
   runtime-efficient (Rust, Haskell, OCaml), wrapping the happy path
   too is correct.* In Python, it isn't — the happy-path wrapper
   earns nothing at runtime and costs caller ergonomics.

5. *=dict.get=-style fallback API* when the caller's intent is
   genuinely "just give me something" — caller supplies the default,
   no type-level branch at all.

6. *=next(genexp, default)= is the "first match" idiom.* When
   combined with a named absence type as the default, the body is
   one line and there is no exception handling anywhere in the
   function.

7. *=None= as an input default* is tolerated only when it genuinely
   means "caller didn't specify and no concrete default fits"
   (extends rule 4 from snippet 001).

*** Still open

- *Type-hint scope* — still deferred from 001.
- *Where named absence types live* — =NotFound= is defined in the
  same file as =find_first=. In a multi-module project, is there a
  shared catalog of absence types, or does each module define its
  own? Deferred to multi-file stage.

** TODO 003 — =003-point.py= — dataclass with a method

Presented version:

#+begin_src python
from dataclasses import dataclass
from math import hypot


@dataclass(frozen=True)
class Point:
    x: float
    y: float

    def distance_to(self, other: "Point") -> float:
        return hypot(self.x - other.x, self.y - other.y)
#+end_src

*** Question 1 — Record type: =@dataclass=, =NamedTuple=, or plain class?

Python gives three built-in ways to model a "two floats with a name":

- (a) =@dataclass(frozen=True)= (presented). Immutable, =__eq__= /
  =__hash__= / =__repr__= generated. Attribute access. Supports
  methods naturally. No tuple-unpacking ambiguity (=p.x= is always
  =p.x=, never =p[0]=).
- (b) =typing.NamedTuple= subclass. Immutable by construction, tuple
  semantics (unpack, indexable), ~25% less memory than a dataclass,
  =__eq__= / =__hash__= via tuple. Methods allowed but feel grafted.
  Callers can treat it as a tuple which is sometimes wanted and
  sometimes a leak.
- (c) Plain class with =__init__=, =__eq__=, =__hash__=,
  =__repr__= hand-rolled. Most flexibility, most ceremony. Only
  justified when the class does something the generators can't
  (custom hashing, lazy fields, inheritance games).

*** Question 2 — =frozen=True= by default?

The presented version is =frozen=True=. Frozen dataclasses are
immutable (attribute assignment raises), hashable by default, and
safer as dict keys / set members. Mutable dataclasses are what you
get without =frozen= — =Point(0, 0).x = 1= works, and =__hash__= is
set to =None= unless you opt in.

- (a) Default =frozen=True= for all record/value types; reserve mutable
  dataclasses for things that genuinely own state over time (caches,
  builders).
- (b) Default mutable; add =frozen=True= only when needed for hashing
  or explicit immutability.

*** Question 3 — Methods on a data record

The presented version has =distance_to= as a method. The alternative
is a free function:

#+begin_src python
def distance(a: Point, b: Point) -> float:
    return hypot(a.x - b.x, a.y - b.y)
#+end_src

The Hickey / data-orientation position is that data records should be
passive — just named fields — and behavior lives in free functions
that can be composed, shadowed, and tested independently. Methods
conflate "what this thing is" with "what you can do with it," and
encourage inheritance-driven elaboration that obscures the data.

The OO / encapsulation position is that =p.distance_to(q)= reads
more naturally than =distance(p, q)= and groups related behavior
with the data.

- (a) Methods on records are fine for natural "self-referential"
  behavior (=distance_to=, =translate=, =magnitude=).
- (b) Records are passive; all behavior lives in free functions.
- (c) Records carry only behavior that's inseparable from the type
  (e.g. a custom =__repr__=); everything else is a free function.

*** Question 4 — Forward reference style

=def distance_to(self, other: "Point")= uses a string literal for
the self-referential annotation because =Point= isn't yet defined
when the method signature is parsed. The alternatives:

- (a) Quoted string (presented): =other: "Point"=.
- (b) =from __future__ import annotations= at the top of the module,
  then all annotations are strings at runtime and =other: Point= just
  works.
- (c) Python 3.12+ =Self= from =typing=: =other: Self=.

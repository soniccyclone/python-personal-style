*** Question 1 — Docstring style on small functions

The current version uses a Google-style sectioned docstring:

#+begin_src python
def greet(name: str, greeting: Optional[str] = None) -> str:
    """Return a greeting for the given name.

    Args:
        name: Person to greet.
        greeting: Optional custom salutation. Defaults to "Hello".

    Returns:
        The formatted greeting.
    """
#+end_src

Alternatives:

- (a) *Google sections always*, even on tiny functions (current).
- (b) *One-line docstring* for functions under ~10 lines; sections reserved for nontrivial APIs.
- (c) *No docstring* unless the function is part of a published API or has non-obvious behavior; rely on naming + types.
- (d) *NumPy-style* sectioned docstrings instead of Google.
- (e) *Sphinx =:param:/:returns:=* style.

*** Question 3 — =None= sentinel vs other default idioms

When a function takes an optional argument that needs a runtime default, there are three common patterns:

#+begin_src python
# (a) Sentinel + reassign (current)
def greet(name, greeting=None):
    if greeting is None:
        greeting = "Hello"
    ...

# (b) Literal default in signature
def greet(name, greeting="Hello"):
    ...

# (c) Short-circuit collapse (only works if falsy values are equivalent to missing)
def greet(name, greeting=None):
    greeting = greeting or "Hello"
    ...
#+end_src

(b) is simplest when the default is immutable and cheap.
(a) is necessary when the default is mutable or expensive to compute.
(c) conflates =None= with other falsy values (empty string, =0=) — sometimes wrong.

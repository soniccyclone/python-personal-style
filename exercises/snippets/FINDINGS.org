** DONE 001 — =001-greet.py= — plain function with args and return

File: [[file:001-greet.py][001-greet.py]]

*** Presented (v1)

#+begin_src python
"""Small greeting helper — nitpick fodder for snippet 001."""
from typing import Optional


def greet(name: str, greeting: Optional[str] = None) -> str:
    """Return a greeting for the given name.

    Args:
        name: Person to greet.
        greeting: Optional custom salutation. Defaults to "Hello".

    Returns:
        The formatted greeting.
    """
    if greeting is None:
        greeting = "Hello"
    return f"{greeting}, {name}!"


if __name__ == "__main__":
    print(greet("Nathan"))
    print(greet("Nathan", greeting="Howdy"))
#+end_src

*** Feedback

Nathan: "So this is all very verbose. Especially the docstring, it's
very googley. I really like the way clojure has done things:
[[https://github.com/clojure/clojure/blob/clojure-1.12.4/src/clj/clojure/core_print.clj#L16][clojure/core_print.clj]]"

Reading the referenced file: public vars get short *prose* docstrings
that describe behavior, defaults, and edge cases — no =Args:= /
=Returns:= / =Raises:= templates. Internal helpers
(=print-sequential=, =print-meta=, =print-object=, etc.) carry *no*
docstring at all; name + inline code is the documentation. Names of
well-understood parameters are single-letter (=w=, =o=, =xs=).
Structure is dense; no ceremony.

*** Revised (v2)

#+begin_src python
def greet(name: str, greeting: str = "Hello") -> str:
    return f"{greeting}, {name}!"
#+end_src

*** Rules canonized from this snippet

1. *Docstrings are prose, not templates.* No =Args:= / =Returns:= /
   =Raises:= section headers (Google / NumPy / Sphinx styles are out).
   When a docstring exists, it's one or more complete sentences
   describing what the thing does and any non-obvious behavior.

2. *No docstring on small, self-explanatory functions.* Rely on name +
   type hints. Docstrings are for public API surface and for any
   behavior that can't be read off the signature.

3. *No module docstrings on snippet files.* The filename and context
   already say what the file is.

4. *Literal default values in the signature* when the default is
   immutable and cheap (=greeting: str = "Hello"=). Reserve the
   sentinel-and-reassign pattern (=x=None= then
   =if x is None: x = ...=) for cases where the default is mutable or
   has to be computed at call time.

5. *No =if __name__ == "__main__":= demo block on snippets.* A snippet
   is a thing to read, not a thing to run. Demos belong in tests or a
   REPL, not wedged into the file.

*** Still open

- *Union-with-=None= syntax* — the PEP 604 =str | None= vs
  =Optional[str]= question never got to exercise its muscle because
  the revised snippet has no nullable arg. Deferred to a later snippet.
- *Type-hint scope* — the revised version keeps full annotations on
  both args and the return. Clojure's precedent would argue for
  annotating only where it clarifies (analogous to =^Writer= interop
  hints on specific args). Still a real question: do we annotate all
  public function signatures by default, or only when the types aren't
  obvious from the name? Deferred to a snippet where the choice is
  more consequential.

** DONE 002 — =002-find-first.py= — iteration, sum-type result

File: [[file:002-find-first.py][002-find-first.py]]

*** Presented (v1)

#+begin_src python
def find_first(items: list[str], query: str) -> str | None:
    for item in items:
        if item == query:
            return item
    return None
#+end_src

*** Feedback on v1

Nathan:

- =None | str= ordering inverts the meaning to "usually nothing but
  sometimes str" — misrepresents the function.
- "Architecting around =Optional= like this is already a code smell
  because it's utilizing =None= as an overload of an end flag rather
  than using a proper type that defines that something has ended."
- "If I am going to bother with type hints in a dynamic language I
  want EXPLICIT types that tell the whole story and don't leave
  things up to guesswork like pythonistas (read: googlers) are so
  willing to do."
- Prefers =next()= over a generator for this snippet specifically.

Underlying argument: =None= is already saturated with language-level
meanings (default arg, no-return, uninitialized, cache miss, etc.);
reusing it as a domain "not found" sentinel layers yet another
meaning on an already-overloaded symbol — C's =NULL= doing every
absence-meaning at once.

*** Revised (v2) — wrong turn

#+begin_src python
class NotFound(Exception):
    pass


def find_first(items: list[str], query: str) -> str:
    try:
        return next(x for x in items if x == query)
    except StopIteration:
        raise NotFound(query) from None
#+end_src

*** Feedback on v2

Nathan: "This specific example find-first.py is WRONG. Because it is
overloading python's exception system with an improper
responsibility: signal flagging. It is not an EXCEPTION to not find
something you are searching for, that is a legitimate output and
should be reflected in the type itself, not thrown as an error.
This sort of mistake of overloading an exception system is exactly
what leads people to golang's foolishness of removing exception
systems completely, which is also wrong."

The sharpened principle: exceptions are for *actually exceptional*
conditions (I/O failure, contract violation, programmer error,
unrecoverable state). "The thing I searched for wasn't there" is a
legitimate, design-time-expected outcome of a search. It belongs in
the *type*, not in a thrown exception. Go's answer (delete exceptions
entirely because people abuse them) is also wrong; exceptions are
the right tool for truly exceptional conditions and should be kept
pure for that purpose.

*** Revised (v3) — also wrong turn

#+begin_src python
from dataclasses import dataclass
from typing import Generic, TypeVar

T = TypeVar("T")


@dataclass(frozen=True)
class Found(Generic[T]):
    value: T


@dataclass(frozen=True)
class NotFound:
    query: str


def find_first(items: list[str], query: str) -> Found[str] | NotFound:
    return next((Found(x) for x in items if x == query), NotFound(query))
#+end_src

*** Feedback on v3

Nathan: "This syntax APPEARS elegant at first glance but upon deeper
inspection reveals that we have complicated the CALLER location for
the sake of FAKE TYPES. Which boils down to the fact that we have
complicated code path for the sake of documentation. Which is
extremely incorrect."

Then: "Holy crap dude it's just =str | NotFound= that preserves the
happy path while also structurally informing the caller for the not
found path instead of an opaque None type."

The problem with v3 was wrapping the /happy/ path in =Found=. At
runtime =Found= does nothing — it's a dataclass the interpreter
ignores. But it forces every caller to unwrap on every use, even
callers who just want to pass the value through. That's caller
ceremony bought purely for a type-checker label. This is the
Python-specific critique: in Rust where sum types compile to
efficient tagged unions and the compiler enforces exhaustive matching,
wrapping the happy path pays off. In Python, it doesn't.

*** Revised (v4)

#+begin_src python
from dataclasses import dataclass


@dataclass(frozen=True)
class NotFound:
    query: str


def find_first(items: list[str], query: str) -> str | NotFound:
    return next((x for x in items if x == query), NotFound(query))
#+end_src

- Happy path is plain =str=. Caller who just wants the value uses it
  directly — no =.value= unwrap, no =match= statement, no
  =isinstance= branch.
- Failure path is a named domain type =NotFound=, not =None=. Carries
  the =query= it couldn't find — debuggable, not an opaque flag.
- Caller who /does/ need to branch checks =isinstance(result,
  NotFound)= (or the inverse on =str=). Only the branching caller
  pays; the pass-through caller doesn't.
- Body is one line: =next(gen, default)= with the domain type as the
  default. Zero exception handling.

*** Rules canonized from this snippet

1. *Don't use =None= as a domain absence sentinel.* =None= is already
   saturated with the language's absence meanings; layering a domain
   "not found" meaning on top repeats C's =NULL= mistake.

2. *Don't use exceptions for expected-absence outcomes.* Exceptions
   are for actually-exceptional conditions (I/O failure, contract
   violation, programmer error). "Not found when searching" is a
   routine outcome, not an exception. Go's answer (delete exceptions
   entirely) is also wrong — keep the exception channel pure so it
   retains meaning.

3. *Union with a named absence type keeps the happy path unwrapped.*
   Return =T | NamedFailure= — not =Found[T] | NotFound=. The
   happy-path type is used directly by callers who don't need to
   branch; only branching callers pay the =isinstance= cost.
   =NamedFailure= should carry context (what was searched for, what
   condition failed).

4. *In other languages where sum types are compile-time-enforced and
   runtime-efficient (Rust, Haskell, OCaml), wrapping the happy path
   too is correct.* In Python, it isn't — the happy-path wrapper
   earns nothing at runtime and costs caller ergonomics.

5. *=dict.get=-style fallback API* when the caller's intent is
   genuinely "just give me something" — caller supplies the default,
   no type-level branch at all.

6. *=next(genexp, default)= is the "first match" idiom.* When
   combined with a named absence type as the default, the body is
   one line and there is no exception handling anywhere in the
   function.

7. *=None= as an input default* is tolerated only when it genuinely
   means "caller didn't specify and no concrete default fits"
   (extends rule 4 from snippet 001).

*** Still open

- *Type-hint scope* — still deferred from 001.
- *Where named absence types live* — =NotFound= is defined in the
  same file as =find_first=. In a multi-module project, is there a
  shared catalog of absence types, or does each module define its
  own? Deferred to multi-file stage.

** TODO 003 — =003-point.py= — data record with a related function

File: [[file:003-point.py][003-point.py]]

*** Presented (v1)

#+begin_src python
from dataclasses import dataclass
from math import hypot


@dataclass(frozen=True)
class Point:
    x: float
    y: float

    def distance_to(self, other: "Point") -> float:
        return hypot(self.x - other.x, self.y - other.y)
#+end_src

*** Re-orientation — CLOS is the reference

Nathan: "let's re-orient this question since this is beginning to
get into the python class/type system. So I consider that common
lisp is the one who did class systems right with CLOS:
[[https://lispcookbook.github.io/cl-cookbook/clos.html][CLOS reference]]"

CLOS principles that matter here:

- *Generic functions, not methods-on-classes.* =defgeneric= /
  =defmethod= define behavior *outside* the class. =defclass= is a
  pure slot specification.
- *Multi-argument dispatch.* Methods specialize on the types of
  multiple arguments, not just the receiver. No privileged =self=.
- *Data and behavior are separate concerns.* The philosophical
  load-bearing piece, aligning directly with Hickey / Clojure
  data-orientation.

Python translation (what we can and can't do):

- Records as pure data → frozen dataclass with only fields. Matches
  =defclass= as a slot spec.
- Behavior in free functions → =def distance(a, b)= not
  =p.distance_to(q)=.
- Single-argument polymorphism → =functools.singledispatch=.
  Python's stdlib analog to =defgeneric=.
- Multi-argument polymorphism → not in stdlib; =multipledispatch=
  and =plum-dispatch= exist as third-party options. Deferred.
- Some dunders must stay on the class (=__iter__=, =__enter__= /
  =__exit__=, =__post_init__=, etc.) because Python's protocol
  lookup goes through the type. These are a forced exception, not
  an OOP license.

*** Revised (v2)

#+begin_src python
from dataclasses import dataclass
from math import hypot


@dataclass(frozen=True)
class Point:
    x: float
    y: float


def distance(a: Point, b: Point) -> float:
    return hypot(a.x - b.x, a.y - b.y)
#+end_src

Four things happened:

1. =distance_to= moved off =Point= and became a free function
   =distance(a, b)=.
2. The self-referential forward reference (=other: "Point"=) is
   gone because =Point= is already defined at the point where
   =distance= is annotated.
3. =Point= is now a pure slot specification, matching =defclass='s
   role.
4. Dispatch is trivial (two =Point= args); no =singledispatch=
   needed here. A later snippet that exercises actual polymorphism
   (e.g. =area(shape)= over multiple shape types) will surface the
   =@singledispatch= question.

*** Questions remaining (CLOS doesn't directly answer these)

**** Question 1 — When /are/ methods tolerated on a record?

Options:

- (a) Allow only dunders that Python's runtime protocols require;
  every non-dunder behavior is a free function.
- (b) Allow dunders + =__post_init__=-style constructor validation.
- (c) Allow dunders + a small set of "obviously self-referential"
  methods.

Nathan's answer: *(a)*. Only dunders forced by the runtime protocol
(=__iter__=, =__enter__=/=__exit__=, =__post_init__=, customized
=__hash__= / =__eq__= / =__repr__= when the generator can't produce
what's needed). Every non-dunder behavior is a free function.

**** Question 3 — =frozen=True= by default?

Options:

- (a) Default =frozen=True= for records.
- (b) Default mutable; opt in to =frozen=True= when needed.

Nathan's answer: *(a)*. Records are immutable by default. *And:*
"we should clean up the dataclass decorator with a frozen version so
that we aren't slapping that rather verbose decorator all over a
codebase."

The v3 revision below introduces a =record= alias for
=dataclass(frozen=True)=:

#+begin_src python
from dataclasses import dataclass
from functools import partial

record = partial(dataclass, frozen=True)


@record
class Point:
    x: float
    y: float
#+end_src

=partial(dataclass, frozen=True)= returns a callable that, applied
to a class, invokes =dataclass(cls, frozen=True)=. The decorator
form =@record= works directly — no wrapping function needed, no
=@record()= parens-with-no-args ceremony.

In a real project, =record= would live in a shared utility module
(something like =_record.py= or =typing_ext.py=) so it's one import
to every module that needs it.

Sub-question raised: should the =record= alias also set
=slots=True= (Python 3.10+)? =slots=True= drops =__dict__= on
instances, cuts memory, makes attribute access faster, and rejects
typo-assignment to undeclared attributes. It strengthens the CLOS
slot-spec analogy. Nathan's answer: /TBD/.

**** Question 2 — Record type: =@dataclass=, =NamedTuple=, or plain class?

/Deep dive below per Nathan's request./

***** The three options, by capability

- *=@dataclass=* (or our =@record= alias). The stdlib CLOS-=defclass=
  analog. Fields declared with annotations; =__init__=, =__eq__=,
  =__hash__=, =__repr__= generated; =frozen=True= gives immutability
  and hashability; =slots=True= (3.10+) drops =__dict__=;
  =__match_args__= generated for pattern matching. Supports
  defaults, =default_factory=, inheritance (with some field-order
  rules). Full =__post_init__= hook for validation.

- *=typing.NamedTuple= subclass.*

  #+begin_src python
    class Point(NamedTuple):
        x: float
        y: float
  #+end_src

  Immutable by construction, hashable, lower memory. Attribute
  access /and/ tuple semantics: =p.x=, =p[0]=, =a, b = p= all work.
  Has =_replace()= for "modified copy" without =dataclasses.replace=.
  Generated =__eq__= and =__hash__= via tuple.

- *Plain class* with hand-rolled =__init__=, =__eq__=, =__hash__=,
  =__repr__=. Maximum flexibility, maximum ceremony, maximum
  opportunity to forget a dunder and silently break equality or
  hashing.

***** The decisive issue: NamedTuple erases domain identity

The critical property of =NamedTuple= — and the reason it's wrong
under CLOS framing — is that *it is a tuple*. Literally:

#+begin_src python
>>> class Point(NamedTuple):
...     x: float
...     y: float
>>> Point(0, 0) == (0, 0)
True
>>> isinstance(Point(0, 0), tuple)
True
>>> p = Point(1, 2)
>>> p[0], p[1]
(1.0, 2.0)
#+end_src

A =NamedTuple='s domain identity is collapsed into its structural
representation. =Point= isn't a distinct type from the perspective
of equality or structure — it's "a tuple with named accessors."
Anywhere a tuple is accepted, a =NamedTuple= sneaks through;
anywhere a =NamedTuple= is expected, a raw tuple may match.

This is the same category of error that rejected =T | None= in
snippet 002 — a named domain thing collapsed into a lower-level
generic thing. CLOS's =defclass= explicitly does /not/ do this;
=defclass='d slots are not a list. The Python analog that preserves
domain identity is =@dataclass= (→ =@record=), not =NamedTuple=.

Corollary issues:

- Callers can index: =p[0]= works but has no field-name discipline
  and rots when fields are reordered.
- Callers can do positional unpacking: =a, b = p= works for two-field
  records; the moment someone adds a field, every unpacking site
  breaks silently or noisily depending on the unpack form.
- No =__post_init__= — no clean validation hook.
- Inheritance between NamedTuples is awkward (can't extend fields).

The performance story (~25% less memory, faster construction in
tight loops) is real but irrelevant to style. If you're in a hot
path where those savings matter, reach for a specific tool. For
general "named record" modeling, identity matters more than bytes.

***** Plain class isn't in the records picture

Nathan: "why even use a plain class for dunders? Can't dunders still
be put onto our record class type when we truly need them? Why ever
switch back to plain?"

Correct — the plain-class escape hatch I'd listed earlier was wrong.
Walking through the cases I'd flagged:

- *Custom =__hash__= over a subset of fields.* Just define it in the
  =@record= body; dataclass preserves user-defined dunders. If you
  also customize =__eq__=, pass =eq=False= to the dataclass so its
  generator doesn't collide. Still a =@record=, still frozen, still
  gets =__init__= and =__repr__= for free.
- *"Computed fields" via =@property=.* These don't belong on the
  record at all. =@property= is a method wearing attribute clothing;
  per Q1's canonized stance, computed values are free functions.
- *Identity-based equality.* =@dataclass(frozen=True, eq=False)=
  inherits =object.__eq__= (identity) and =object.__hash__=
  (id-based). No plain class needed.
- *Lifecycle objects with state-and-behavior over time* — file
  handles, connections, caches. These /aren't records/. They're a
  different category entirely: mutable, identity-equality, nontrivial
  =__init__=, field-based =__eq__= / =__repr__= makes no sense.
  Plain class territory, but outside the records question.

*Net:* for records there is no "fall back to plain class." The only
question is =@record= versus =NamedTuple=, and =NamedTuple= is out
for the identity-erasure reason above.

Plain classes exist for a different category: non-record things
with lifecycle. That distinction deserves its own rule — not a
branch of the records question.

***** Nathan's answer (Q2): =@record= for records, full stop

- =@record= for any "named thing with typed fields."
- =NamedTuple= only when tuple semantics are /actually wanted/ —
  genuinely positional data (CSV rows, matrix coordinates) where
  indexing and unpacking are part of the intended interface. Narrow
  and domain-specific.
- *Plain classes are not an option for records.* Dunders that the
  runtime protocol forces onto the type go directly on =@record=,
  with dataclass flags (=eq=False=, =unsafe_hash=True=, etc.)
  adjusted as needed.
- Plain classes /are/ the right tool for non-records — lifecycle
  objects where identity, mutability, and custom =__init__= logic
  are the point. Separate question, separate rule.

*** Rule

Q1, Q2, and Q3 canonized. Sub-question on =slots=True= still open.
Q4 (polymorphism mechanism) deferred to the next snippet.

Canonized rules (see [[file:../../README.org][README.org]] /Records & Behavior/):

1. Records are pure data. Fields only; behavior is free functions.
2. Only runtime-protocol dunders are tolerated on records
   (=__iter__=, =__enter__= / =__exit__=, =__post_init__=, customized
   =__hash__= / =__eq__= / =__repr__= when generators can't produce
   what's needed). Non-dunder methods and =@property= → free functions.
3. =@record= = =partial(dataclass, frozen=True)= is the canonical
   record decorator. In a real project it lives in a shared utility
   module.
4. Records are frozen by default.
5. =NamedTuple= is out for records — it IS a tuple and so erases
   domain identity. Reserve for positional data where tuple
   semantics are intentional.
6. Plain classes are not part of the records question. They're for
   lifecycle objects (state + behavior over time, identity equality,
   custom =__init__=). Separate category, separate rule.

* About

A literate-programming style guide for Python, derived by nitpicking.

This README is the *final, compiled* takeaway. It is populated progressively
as the exercises under [[file:exercises/LEDGER.org][exercises/]] uncover concrete rules.

PEP 8 is the starting point; everything here is either a deviation from, or a
sharpening of, PEP 8 based on lived preference.

* References

This style guide is anchored by three references, in precedence order:

1. *Clojure core* — [[https://github.com/clojure/clojure/blob/clojure-1.12.4/src/clj/clojure/core_print.clj][clojure/core_print.clj]]. Taste anchor for
   terseness, prose-not-template docstrings, docstrings only on
   public API, dense ceremony-free code.
2. *CLOS* (Common Lisp Object System) — [[https://lispcookbook.github.io/cl-cookbook/clos.html][cl-cookbook/clos.html]]. Class
   system anchor for data/behavior separation, =defclass= as slot
   spec, generic functions, polymorphism via dispatch-on-types.
3. *Barry Warsaw's style guide* — [[https://barry.warsaw.us/software/STYLEGUIDE.txt][barry.warsaw.us/software/STYLEGUIDE.txt]].
   Python-specific mechanics — imports, line length, quotes, etc.
   Subordinate to the Lisp-family references; where Warsaw conflicts
   with a rule derived from Clojure or CLOS, the Lisp-family rule
   wins.

PEP 8 is the implicit floor: anything not overridden by the three
references above follows PEP 8.

** Imports

Reference: Barry Warsaw's [[https://barry.warsaw.us/software/STYLEGUIDE.txt][STYLEGUIDE.txt]].

- *=__future__= imports go at the very top*, above every other
  import. Standalone, no blank line before them.
- *Four groups, separated by blank lines*, in this order:
  1. Non-from imports from stdlib and third-party (=import json=).
  2. Non-from imports from the application (local).
  3. From-imports from stdlib and third-party
     (=from pathlib import Path=).
  4. From-imports from the application (local).

  The primary axis is import /style/ (=import X= vs =from X import
  Y=), not just origin — more granular than PEP 8's three-group
  scheme. Rationale: visual uniformity per group.
- *Dotted imports follow non-dotted within a group.*
  =import os= before =import os.path=.
- *Non-dotted imports sort by increasing length*: =import os= before
  =import json= before =import logging=. Deliberately visual — short
  lines stack neatly, longer lines below.
- *Dotted and from-imports sort alphabetically.*
- *Two blank lines between the import block and module-level code.*

The Warsaw-canonical ruff config that enforces all of this:

#+begin_src toml
  [tool.ruff.lint.isort]
  case-sensitive = true
  length-sort-straight = true
  lines-after-imports = 2
  lines-between-types = 1
  order-by-type = true
  section-order = ['standard-library', 'third-party', 'local-folder', 'first-party']
#+end_src

** Records & Behavior

Reference: CLOS ([[https://lispcookbook.github.io/cl-cookbook/clos.html][cl-cookbook/clos.html]]). =defclass= is a pure slot
specification; behavior lives in =defgeneric= / =defmethod= outside
the class, dispatching on the types of (potentially multiple)
arguments. Python translation below.

- *Records are pure data.* Fields only; no methods, no properties.
  Behavior lives in module-level functions that take the record as
  an argument.
- *Only runtime-protocol dunders are tolerated on records.*
  =__iter__=, =__enter__= / =__exit__=, =__post_init__=, customized
  =__hash__= / =__eq__= / =__repr__= when the generated versions
  aren't right — things Python's runtime looks up through the type
  and that can't be expressed as free functions. Non-dunder
  "methods" and =@property= are free functions.
- *Records are frozen and slotted by default, via a =record= alias.*
  Don't scatter =@dataclass(frozen=True, slots=True)= everywhere —
  define it once in a shared utility module and import:

  #+begin_src python
    from dataclasses import dataclass
    from functools import partial

    record = partial(dataclass, frozen=True, slots=True)
  #+end_src

  then at each declaration site:

  #+begin_src python
    @record
    class Point:
        x: float
        y: float
  #+end_src

  =slots=True= closes the object layout — what's declared is what
  exists, typo assignment (=p.xx = 1=) raises, and the per-instance
  memory footprint drops from ~200 bytes to ~72 for a two-field
  record. This is the CLOS =defclass= slot-spec semantic in Python.
  =__weakref__= is /not/ in the default slot list; if a specific
  record type genuinely needs to be weakly referenceable, that class
  overrides the alias with =@dataclass(frozen=True, slots=True,
  weakref_slot=True)= explicitly — opt-in, visible at the
  declaration site where it matters.

- *Dunders needing unusual semantics go directly on the =@record=
  class body.* Adjust dataclass flags as needed (=eq=False= if you
  override =__eq__=, =unsafe_hash=True= if you want a custom
  =__hash__= to coexist with generated =__eq__=, etc.). There is no
  "fall back to plain class" for records — if it's a record, it's
  =@record=.
- *Don't use =typing.NamedTuple= for records.* A =NamedTuple= /is/ a
  tuple: =Point(0, 0) == (0, 0)= returns =True=, =p[0]= works,
  =isinstance(p, tuple)= is =True=. Domain identity collapses into
  structural representation. Same category of error as =T | None=
  overloading =None=. Reserve =NamedTuple= for cases where tuple
  semantics are /intentionally/ part of the interface — genuinely
  positional data like CSV rows or matrix coordinates.
- *Plain classes are for non-records.* State-plus-behavior objects
  with a lifecycle (file handles, connections, caches) — mutable,
  identity-equality, nontrivial =__init__=, field-based
  =__eq__= / =__repr__= would be wrong. Those are a separate
  category from records; the rules above don't constrain them.

*Polymorphism over records* follows CLOS:

- *=@functools.singledispatch= is the polymorphism mechanism.* Stdlib.
  Closest Python analog to =defgeneric= / =defmethod=. No third-party
  multi-dispatch for single-arg cases; no =isinstance= branching; no
  ABC with overridden methods.

  #+begin_src python
    from functools import singledispatch

    @singledispatch
    def area(shape: object) -> float:
        raise NotImplementedError(f"area for {type(shape).__name__}")

    @area.register
    def _(shape: Circle) -> float:
        return pi * shape.radius ** 2

    @area.register
    def _(shape: Square) -> float:
        return shape.side ** 2
  #+end_src

- *Registered implementations use =_= as the name.* Not
  =area_circle=, not the generic name re-used. =_= prevents
  importable module-level bypass paths — =from shapes import
  area_circle= with a descriptive name lets callers skip the
  dispatch entirely, which defeats the polymorphism. With =_=, only
  the last =_= is reachable at module scope (each =def _= shadows
  the previous) and the leading underscore is Python's "private"
  signal; =area(shape)= becomes the only usable entry point.
- *The base function raises =NotImplementedError= on unregistered
  types*, with a message naming the unknown type:
  =raise NotImplementedError(f"{fn} for {type(x).__name__}")=.
  Dispatch failure is a programmer error, genuinely exceptional;
  exceptions are the right tool. Stdlib convention
  (=NotImplementedError= as the marker for un-overridden polymorphic
  stubs) agrees.
- *No shared interface marker for participating records.* Records
  stay independent; the binding is the set of registered methods
  on the generic function. No =Shape= Protocol, no =Shape= ABC. A
  local union alias is fine as a downstream typing convenience
  (=def total_area(shapes: list[Shape])=) but doesn't flow from the
  dispatch pattern itself.
- *The Pylance reach-ability gap is accepted.* Pyright's typeshed
  stub for =singledispatch= types the wrapper as
  =(*args: Any, **kwargs: Any) -> _T= — so =area(42)= passes
  type-checking and fails at runtime with =NotImplementedError=. No
  =@overload= declarations to close the gap (ceremony that creates
  importable bypass vectors and triples the touchpoints per new
  dispatch); no =.pyi= stub generation (working around an IDE
  limitation with a build-time ecosystem is the wrong shape of
  response). Pyright is a static-only LSP with no plugin system;
  dynamic dispatch is outside its scope by design. Runtime
  =NotImplementedError= is the backstop. Navigate via find-references
  on the record type; introspect at runtime via =fn.registry=.

** Types & Annotations

- *Return types tell the whole story.* A function's return annotation
  describes /what the function produces/, not a riddle the caller has
  to solve. If the function can fail to produce a value, the failure
  must be a *named* domain type, not =None= sitting inside a union.
  See Errors & Exceptions for the =T | NamedFailure= shape.

** Errors & Exceptions

- *Exceptions are for exceptional conditions, not control flow.*
  Programmer errors, I/O failures, contract violations, unrecoverable
  state. "The thing I searched for wasn't there" is /not/
  exceptional — it's a legitimate, design-time-expected outcome of a
  search, and the function's type must say so. Go's response (remove
  exceptions entirely because people abuse them) is also wrong:
  exceptions are the right tool for actually-exceptional conditions,
  and throwing them out to discipline misuse throws out the real
  utility too.
- *Don't overload =None= as a domain absence sentinel.* =None= is
  already saturated with language-level meanings (default argument,
  no-return, "not initialized", cache miss, etc.); adding yet another
  domain meaning on top repeats C's =NULL= mistake at the Python
  level.
- *Return =T | NamedFailure= for expected failure modes.* Keep the
  happy-path type unwrapped — callers who don't need to branch use
  the =T= directly — and name the failure branch with a frozen
  dataclass that carries context:

  #+begin_src python
    @dataclass(frozen=True)
    class NotFound:
        query: str

    def find_first(items: list[str], query: str) -> str | NotFound:
        return next((x for x in items if x == query), NotFound(query))
  #+end_src

  Do *not* wrap the happy path too (=Found[T] | NotFound=) — in
  Python that's ceremony without payoff. The wrapper does nothing
  at runtime but forces every caller to unwrap on every use.
  (In Rust / Haskell / OCaml where sum types are compile-time
  enforced and runtime-efficient, wrapping both sides /is/ the
  correct shape. Python isn't that language; don't pretend it is.)
- *=dict.get=-style fallback API* when the caller wants an inline
  default value — caller supplies the default, no type-level branch
  at all. Cleaner than either =T | None= or =T | NamedFailure= when
  the caller's intent is "just give me *something*."
- *=raise X from None=* when an exception /is/ the right tool and
  we're translating an implementation-detail exception into a domain
  one — keeps tracebacks clean.

** Control Flow

- *Literal default in the signature* when the default is immutable
  and cheap (e.g. =greeting: str = "Hello"=). Reserve the
  =x=None= sentinel + reassignment pattern for cases where the
  default is mutable or has to be computed at call time.
- *=None= as an input default* is tolerated only when it genuinely
  means "caller didn't specify and no concrete default fits." When a
  concrete default exists, use it.
- *=next(genexp, default)= is the "first match" idiom.* Put the
  named-failure constructor directly in the default slot:
  =next((x for x in items if x == query), NotFound(query))= — body
  is one line, no exception handling, no sentinel.

** Comments & Docstrings

- *Docstrings are prose, not templates.* No =Args:=, =Returns:=, or
  =Raises:= section headers (Google / NumPy / Sphinx styles are all
  rejected). Write one or more direct sentences describing behavior
  and any non-obvious constraints. Model: Clojure core, e.g.
  [[https://github.com/clojure/clojure/blob/clojure-1.12.4/src/clj/clojure/core_print.clj#L16][=*print-length*=]].
- *No docstring on small, self-explanatory functions.* Rely on the
  name and type hints. Docstrings earn their place on public API
  surface and on behavior that can't be read off the signature.
- *No module docstring unless the module exports a coherent public
  API that needs one.* The filename and directory already tell you
  what the file is.
- *Demo blocks don't belong in source files.* If you need to show
  usage, put it in a test or the REPL — not in
  =if __name__ == "__main__":= at the bottom of the module.

* Method

The rules below were extracted by working through three graduated exercises:

1. *Snippets* — small, single-concept fragments.
2. *Single-file programs* — complete programs in one module, each with its
   own virtual environment and =pyproject.toml=.
3. *Multi-file programs* — full package layouts, each with its own venv.

At each step Nathan nitpicked the output. Findings were recorded in the
per-category =FINDINGS.org= files, then compiled into the rules above.
